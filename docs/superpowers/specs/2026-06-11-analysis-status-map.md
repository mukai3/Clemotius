# Clemoutis 解析状況マップ

作成日: 2026-06-11
目的: オリジナル（かざぐるマウス）の解析で「何が確定し、何が未確定か」を一覧化し、
今後どこを調べれば実装が進むかを示す。全関数デコンパイルはコスト不見合いのため行わない方針。

## 解析手段の整理

| 手段 | できること | 環境 |
|---|---|---|
| 静的解析（PEインポート/リソース/ヘルプ） | 構造・API・UIラベル・設定形式・選択肢名 | 実施済み（このリポジトリで完結） |
| 逆アセンブル（Ghidra/IDA） | コード値↔意味、アルゴリズム詳細 | 高コスト。狙い撃ちのみ想定 |
| 動的解析（VMで実機観測） | 各操作→ini値/送出メッセージの対応 | **ユーザー実機作業が必要**（GUI操作＋観測） |

## 確定済み（静的解析で判明）

- 構成: `Kazaguru.exe`(本体), `Kazasub.dll`(設定UI=17ダイアログ), `Kazahook*.dll`(フック, shared セクション=DLLインジェクション), `Kazawow64.exe`(32bitブリッジ)
- アクション機構: キー送信(SendInput/keybd_event) ＋ ウィンドウメッセージ(SendMessage/PostMessage/SendNotifyMessage) のハイブリッド。AttachThreadInput+SetForegroundWindow で確実配送
- UI要素判定: OLEACC(MSAA) 使用
- 設定ファイル: UTF-16LE INI（BOM無）。`[Settings]`/`[Gestures]`/`[MouseAssignment]`
- ジェスチャー値の2形式: 整数=組み込みコマンドID / `#1,VK,Shift,Ctrl,Alt,Win`=キー送信
- 設定画面タブ: オプション/ホイール/ボタン/拡張スクロール/マウスジェスチャー/シェル操作/ウィンドウ/詳細
- 拡張スクロール修飾キー: 6種（Shift/Ctrl/Ctrl+Shift/Alt/Shift+Alt/Ctrl+Alt）
- ホイールタブ: 垂直/水平スクロールバー上の挙動が各ドロップダウン
- アプリ種別カテゴリ: IE系/Mozilla系/Opera/Safari系/Chromium系/エクスプローラ/ウィンドウ（Chromiumは独立カテゴリ）
- コマンドカタログの一端（メニューリソースより）: コピー/取得系（選択テキスト・ブラウザURL/タイトル・リンク先URL/テキスト・エクスプローラのフォルダ/選択ファイル・ウィンドウの実行ファイル/タイトル 等）

## 動的解析で確定（2026-06-11、ユーザー提供 ini の差分より）

ユーザーが appdefault / mydefault / modscroll の3 ini を提供。modscroll は appdefault から
「垂直バー=1行スクロール / 水平バー=水平3列スクロール」に変更したもの。差分で以下を確定。

注: appdefault / modscroll は **ANSI(cp932)** エンコード、mydefault は **UTF-16LE**（読み分け必須）。

### ScrollEx* コードの構造
- `appdefault→modscroll`: `ScrollExOnVScrlBar` 53→50、`ScrollExOnHScrlBar` 58→57
- ビット分解（全サンプル整合）: 上位は `0x30` ベース、**bit3(0x08)=方向（1=水平 / 0=垂直）**
  - 57(水平3列)/58(水平既定)→水平、50(垂直1行)/53(垂直既定)→垂直
- 下位3bit=単位/量（57→1, 50→2, 53→5, 58→2）。**量の完全な意味は未確定**（各選択肢を1つずつ変えた ini が追加で必要）
- → **実装反映済み**: `ScrollBehaviorParser` が `code:NN` の bit3 で水平/垂直を判定。これでユーザー既定の `code:58`(水平バー)・Shift+ホイール(58) が水平スクロールとして動く

### スクロールバー上の動作のコード（完全確定）
ユーザー提供 ini（scr-1-1 / scr-3-3 / scr-6-6 = 垂直/水平を 1,1 / 3,3 / 6,6 に設定）で**連番と確定**:

```
垂直バー: 1行=50 3行=51 6行=52 9行=53 12行=54 ページ=55 高速=56 上端下端=57   (= 50+index)
水平バー: 1列=56 3列=57 6列=58 9列=59 12列=60 ページ=61 高速=62 左右端=63   (= 56+index)
```
- 工場出荷時/mydefault 既定: 垂直=**53(9行)** / 水平=**58(水平6列)**（mydefault スクショと一致）
- **重要**: 56/57 は垂直(高速/上端下端)と水平(1列/3列)で値が重複する。本来の方向は「どちらのスロット
  (垂直バー用/水平バー用)か」で決まり、**コード単体では方向を一意に決められない**。
  → 以前の「bit3=方向」仮説は誤り（垂直の高速56/上端下端57で bit3 が立つため破綻）。
- 実装（反映済み）: `ScrollBehaviorParser` はコード値域（56以上＝水平系）で近似判定。既定値は意味が
  判明したので名前付き（`onVerticalScrollbar="none"` / `onHorizontalScrollbar="horizontal"`）に置換。
- 注: modifier+ホイールのコードも同体系（ScrollExShift=58=水平6列→水平化、ScrollExCtrl=53=9行→垂直）。
  「なし」選択時のコード値のみ未サンプル（Clemoutis は "none" 文字列で表現するため実害なし）。

### 工場出荷時（appdefault）の組み込みコマンドID
`L=1`(戻る) `R=2`(進む) `UD=3` `DR=6`(閉じる系) `R+WU=110`(前のタブ) `R+WD=111`(次のタブ)。
mydefault 差分より `110`=前のタブ/`111`=次のタブが確定、`DR=6` はユーザーが Ctrl+W に置換＝閉じる系と判明。
未確定の組み込みID: `3`(UD), `5`(U), `101`(LR), `112`(DL)。

## 未確定（残り）

| 項目 | 取り方 | どの実装に効くか | 優先度 |
|---|---|---|---|
| 組み込みID `3`/`5`/`101`/`112` の意味 | 動的: 各コマンドを割当→実行して送出を観測 | ジェスチャーのアクション拡張 | 中 |
| 各アクションの実送出機構 | 動的: Spy++/API Monitor | 挙動一致の受け入れ基準 | 中 |
| 標準スクロールバーの厳密判定 | 逆アセンブル/MSAA実装 | ブラウザ等カスタム描画スクロールバーの検出 | 中 |
| スクロールバー「なし」のコード値 | 動的: ini で「なし」を選んで採取 | 任意（Clemoutis は "none" 文字列で代替済み） | 低 |

## 方針

- 重い一括解析（全関数デコンパイル）は行わない。言語/構造が別物（C++ネイティブ→C#）で移植に直結せず、コスト過大・法的にもグレーが濃い。
- 必要時に **狙い撃ち**で解析。特に「高」優先の `ScrollEx*` コードは、ユーザーが実機でオリジナルを操作できるタイミングで動的解析するのが最短。
- それまでは未確定コードを `code:NN` として保持（素通し）し、使用感を壊さない。
